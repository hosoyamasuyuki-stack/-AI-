## プロジェクトの目的
================================================================

日経225比年率+3.9%超過リターンを継続達成（上位3〜4%水準）。
10年以上の投資経験を持つ個人投資家（細矢さん）が運用する自己学習型AI投資判断システム。

数値目標：
- 日経比年率+3.9%超過リターン（日本投資家上位3-4%相当）
- 2億円資産 x 年間+780万円のインデックス超過

ベンチマーク定義：
- 比較対象：TOPIX総収益 vs 上位3%アクティブファンド
- アルファ計算：株式リターン - ベータ x 市場リターン

4軸予測システム：
- 目先（3ヶ月）：エントリー・エグジットタイミング
- 短期（1年）
- 中期（3年）
- 長期（3年以上・無期限・永久保有候補）

重要な認識：
- 1年判断は勝率50%（コイン投げ）→長期保有が有効
- 5年保有：買いシグナル銘柄88%が日経を上回ることを確認済み
- バックテスト修正①：5Y勝率100%・超過+287%

将来計画：
- WEB上で完結するシステムとして販売予定
- エンジニアに渡して引き継ぐ予定
- GitHubを唯一の真実の源として管理

個人投資家としての構造的優位性：
1. 小型株へのアクセス（機関投資家は大量資金で投資不可）
2. 迅速な意思決定（組織承認不要）
3. 10年以上の経験的判断（AIが定量・人間が定性）
4. 5年保有という時間的優位性（1年は50%・5年は88%超過）

## スコアリングモデル詳細（v4.2確定版）
================================================================

総合スコア = Real ROIC(40%) x トレンド(35%) x 価格(25%)

変数1：Real ROIC（40%）
= ROEスコア x 60% + FCF変換率スコア x 40%
ROE基準：25%以上→100pt / 20%→85pt / 12%→58pt / 5%以下→20pt
FCR基準：80%以上→78pt以上 / 60%→62pt / 20%以下→26pt

変数2：トレンド（35%）
= ROEの3-4年傾き x 60% + FCR傾き x 40%
傾き基準：+4%/年以上→100pt / +2%→82pt / 0%→64pt / -2%→28pt

変数3：価格（25%）
= PEGスコア x 50% + FCF利回りスコア x 50%
PEG基準：0.5以下→100pt / 1.0→72pt / 1.5→42pt / 2.0超→26pt
FCF利回り：8%以上→100pt / 4%→70pt / 2%→38pt / 0%以下→8pt

ランク：S>=80 / A>=65 / B>=50 / C<50 / D=低スコア

## 重要な技術的知見（教訓）
================================================================

### 教訓1：PythonからHTMLにJSを埋め込む際の鉄則
- 日本語・絵文字・\u{XXXXX}形式は使わない
- r-string（r'''...'''）内でも\uはPythonが解釈する
- 解決策：バイト列（b'''...'''）を使ってHTMLに直接挿入する
- または：JSを別ファイルにしてscript srcで読み込む（ただし順序問題に注意）

### 教訓2：外部JSファイルの読み込み順序問題
- <script src="kenja.js">をstyleの前に置いても
  インラインscriptが実行される前にkenja.jsの関数が使える保証はない
- 最も確実なのはインラインJSとして同じscriptタグ内に書くこと
- deferやDOMContentLoadedを使う方法もあるが複雑になる

### 教訓3：Nodeによる構文チェックの罠
- node -c でwindow/documentが未定義エラーが出てもブラウザでは正常
- ブラウザで直接fetch→evalして確認するのが正確

### 教訓4：GitHub Pagesのキャッシュ
- URLに?v=Nをつけてキャッシュを回避する
- raw.githubusercontent.comはCDNキャッシュ10分以上の遅延あり
- コミット確認はAPI（api.github.com）を使う
  URL: https://api.github.com/repos/hosoyamasuyuki-stack/-AI-/contents/ai_dashboard_v13.html

### 教訓5：実データ確認前に設計・断言しない
- 存在しないシート名を自信を持って述べない
- コードで参照する前にシート名を確認すること
- 「問題なし」と判断する前に必ず細矢さんの画像と数値を両方確認する

### 教訓6：指示の遵守
- 指示「手順とコードを提出」→余計な操作をしない
- 原因が特定できていない状態で「直ります」と言わない

### 教訓7：Colabコードの注意
- Colabコードには説明文・全角記号を混入させない（SyntaxErrorの原因）

### 教訓8：predict_recordシートの構造
- 3行構造（ヘッダー/サブヘッダー/データ）
- data=rows[2:]が正しい（rows[1:]はサブヘッダーを含むため誤り）

### 教訓9：GitHubトークンの種類
- GitHubブラウザJS書き込みはClassicトークンが必要
- Fine-grainedトークンはCORSでPUT操作がブロックされる

### 教訓10：バックテストのlook-ahead bias
- 「現在スコアで過去リターン検証」は先読みバイアスがある

### 教訓11：書記官の更新ルール
- 更新前に必ず前版を取得・文字数確認してから上書きする
- 新版文字数 >= 旧版文字数 を必ず守る

### 教訓12：推測で作業しない（2026/03/30 全プロジェクト共通）
J-Quants APIキー問題で3回原因を誤診し2日間を浪費した教訓。
- エラー発生時のプロセス：
  1. **まず公式ドキュメントを読む**（5分で済む）
  2. **curl等で実データを使って手動テスト**（1分で済む）
  3. **原因を確定してから実装する**（確定前にコードを書かない）
  4. **調査結果を即座に記録する**（メモリ+CLAUDE.md）
- APIキー問題の鉄則：
  - エラーが出たらまずcurlで手動テスト（3秒で有効性確認可能）
  - 「invalid or expired」→ キーの値自体が間違い。ロジックの問題ではない
  - 複数箇所にキーを設定する場合、どこが古いか全箇所確認
  - 推測で複雑な仕組み（自動リフレッシュ等）を入れない。まず単純な原因を潰す
- 記録の鉄則：
  - 推測段階ではなく**確定後に**記録する
  - 仕様情報は**公式ドキュメントの内容のみ**記録する（推測を事実として書かない）
  - 障害発生時は原因・誤診経緯・対策・再発防止を全て記録する

## 投資判断の思考モデル（2026/03/27全チーム合意・詳細論理記録）
================================================================

### 1. 根本原則：長期保有（10年）が構造的優位性の源泉

論拠：
- H001-Cバックテスト：1年判断の勝率50%（コイン投げ）→ 短期売買は無意味
- H001-C：5年保有で勝率100%・超過+287% → 時間が味方
- H001-C：3年保有で年率+4.46%超過 → 目標+3.9%を上回る
- 結論：保有期間が長いほどv4.3スコアの予測力が発揮される

個人投資家の構造的優位性（機関投資家にない武器）：
1. 小型株アクセス（機関は大量資金で投資不可）
2. 迅速な意思決定（組織承認不要）
3. 5年保有という時間的優位性（機関は四半期ごとに成績評価される）
4. 10年以上の経験的判断（AIが定量・人間が定性）

### 2. 売り判断の廃止 → 四半期レビュー+入れ替え候補制

設計思想：
- 「売り」という行為は感情を誘発する → パニック売りの原因
- 代わりに「入れ替え」という概念を使う → 「より良いものに交換」の心理
- 日次・週次では行動しない → 四半期（3ヶ月ごと）にだけレビュー

ランク変動と行動ルール：
| 状態 | アクション | 頻度 |
|------|-----------|------|
| S/Aランク維持 | 何もしない | - |
| A→B転落 | 何もしない（通常変動） | - |
| B→C転落 | 「注目」フラグ表示 | 週次自動 |
| C→D転落 | 「四半期レビュー候補」リストに追加 | 週次自動 |
| D維持2四半期連続 | 「入れ替え候補」として提示 | 四半期レビュー時 |
| 不正会計/上場廃止リスク | 即時アラート（唯一の例外） | EDINET監視 |

四半期レビュースケジュール：
- 1月・4月・7月・10月の第1週に実施
- レビュー対象：D転落銘柄 + D維持2四半期銘柄
- 入れ替え候補：スクリーニングTop50のSランク銘柄
- 人間の判断：「入れ替えるかしないか」のみ（感情的余地を最小化）

### 3. MacroPhase判定と行動ルール

現在の4層100点スコアリング：
- Layer A リスク指標(40pt)：VIX/HYG/TED/信用スプレッド
- Layer B 金融政策(30pt)：FF金利/日銀/長短金利差/M2
- Layer C 経済活動(20pt)：ISM/失業率/鉱工業生産/設備稼働率
- Layer D バリュエーション(10pt)：CAPE/PBR/益回り/バフェット指標

Phase判定（daily_update.py 行424で実装済み）：
- GREEN（60点以上）：新規買い可能・入れ替え実行可
- YELLOW（30-59点）：選別買いのみ（Sランク限定）・入れ替えは慎重に
- RED（30点未満）：新規買い禁止（現保有は継続保有）・入れ替えも見送り

現在：マクロスコア45点（YELLOW）だがVIX=26.78でリスク指標RED
→ VIXが25以下に戻るまでは実質RED運用

転換条件（明文化 2026/03/27）：
  RED→YELLOW：マクロ総合30点以上 かつ VIX<25 かつ HYG反発（前月比+）
  YELLOW→GREEN：マクロ総合60点以上 かつ VIX<20 かつ 長短金利差>0
  GREEN→YELLOW：マクロ総合59点以下 または VIX>25
  YELLOW→RED：マクロ総合30点未満 または VIX>30

行動ルール：
  RED時：何もしない。保有は継続。新規買い・入れ替え禁止
  YELLOW時：スクリーニングTop50のSランクのみ検討可。四半期レビューで判断
  GREEN時：監視銘柄から積極的に追加可。入れ替え実行可

### 6. 賢者の審判：ディープ・インサイト分析の設計思想

目的：
v4.3スコアが「今買うべきか」を判定するのに対し、賢者の審判は
「今この会社で何が起きているか」を初心者にもわかる言葉で説明する。
予測ツールではなく、現状把握ツール。

設計根拠：
- v4.3スコアは過去の財務データの蓄積（週次更新・タイムラグあり）
- 直近の決算で売上急落・主力事業赤字転落・CF急悪化が起きていても
  スコアに反映されるまでラグがある
- 賢者の審判がそのラグを補完する「今この瞬間の異変検知」

7セクション分析フレームワーク（Deep Insight v3）：
1. 業績の正体 — 売上・利益の増減とその真因
2. 成長の質 — 何で伸びているか（新製品/値上げ/数量/M&A/為替）
3. 持続力 — 構造的成長か一過性か、外部依存度
4. 未来への展望 — 公式予想・経営陣の温度感・予想の透明度
5. リスクの特定 — 上位2-3リスクと対策の有無
6. 資金の血流 — 営業CF・投資CF・フリーCF・財務健全性
7. 最終審判 — 「勝ち筋」は信頼できるか + スコア乖離アラート

alert機能：
v4.3スコアと直近の実態に乖離がある場合、赤い警告を表示。
例：スコアAだが直近決算でCFが急悪化 → 「スコアとの乖離を検知」

技術構成：
ダッシュボード「Analyze」ボタン
→ GASプロキシ（gas_kenja_proxy.js）
→ EDINET API（直近90日の開示書類検索）
→ OpenAI GPT-4o（7セクション分析+JSON出力）
→ ダッシュボードにビジュアル表示（Part A）+ 詳細レポート（Part B折りたたみ）

GASスクリプトプロパティ：
- kenja-rich-api：OpenAI APIキー
- EDINET_API_KEY：EDINET APIキー

初心者配慮：
- 全ラベルを平易な日本語に（「キャッシュフロー」→「お金の流れ（現金は残っているか）」）
- 専門用語には必ず括弧内に説明を添える
- 「初心者向けアドバイス」を末尾に表示
- 統計的根拠のないConfidence値は削除済み（誤解防止）

重要な認識：
- 賢者の審判は「予測」ではなく「現状把握」
- H001-Cで証明済み：1年判断の勝率は50%（コイン投げ）
- どれだけ優れた分析でも1年先の株価は当てられない
- 賢者の審判の価値は「買う前に異変がないか確認する」こと
